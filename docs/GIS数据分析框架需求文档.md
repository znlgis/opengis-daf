# GIS 数据分析与质检框架需求文档

> 对应设计文档：[/docs/GIS数据分析框架设计文档.md](./GIS数据分析框架设计文档.md)

## 1. 文档说明

本文档聚焦框架的业务目标、范围、核心概念、能力需求和约束条件，用于指导后续架构设计、开发排期和验收。  
关于技术实现细节、接口模型和执行机制，请参阅配套的《GIS数据分析框架设计文档》。

## 2. 项目背景

本项目旨在构建统一的 GIS 数据分析与数据质检框架。  
数据分析与数据质检在本质上都围绕“配置方案、执行处理、产出结果”展开，因此框架需要在一套统一能力底座之上同时支撑分析场景和质检场景，避免重复建设。

## 3. 建设目标

### 3.1 总体目标

- 建立统一的 GIS 数据分析与质检能力底座
- 通过配置化方案支持不同业务流程快速组合
- 支持多种数据源、多种输出方式和可扩展算子体系
- 为后续服务化、平台化和自动化集成打下基础

### 3.2 核心理念

无论是数据分析还是数据质检，其最根本的执行单元都可以抽象为一个原子化模型：**输入数据 → 执行操作 → 产出结果**。一个算子（Operator）就是这样一个原子操作的封装——接收输入、执行逻辑、返回输出。

当需要同时处理一组数据或串联一系列操作时，仅仅有孤立的算子是不够的，这时便产生了"方案"的概念。方案是多个算子实例化后（即分析项）按一定执行关系组织起来的有序集合，经框架统一调度执行后产出一个完整的业务结果（如质检报告、分析结果集、统计报表等）。

分析场景与质检场景之间的本质关系是理解本框架的关键：**质检是分析的一种特化场景**。二者的核心差异不在于技术执行层面，而在于关注点的不同——分析重在"结果是什么"，质检重在"过程哪里有问题"。因此，框架只需在一套执行引擎之上，为质检场景额外叠加过程追溯、问题清单和中间结果保留等能力即可。

基于以上认识，框架遵循以下核心理念：

- **方案驱动**：以分析方案为核心组织业务流程，方案可配置、可复用，支持通过子方案引用的方式实现分层组合（子方案嵌套机制将在后续迭代中完善，详见 5.3(2)）
- **统一框架**：分析和质检共用同一套执行框架，通过算子复用和输出适配，避免为两种场景分别建设独立系统
- **能力扩展**：通过算子体系实现能力扩展，算子需遵循统一接口，支持插件式注册和自定义扩展
- **过程可追溯**：尤其在质检场景下，保留过程信息、中间结果和执行日志，支撑问题溯源和结果复核

## 4. 应用场景

| 场景类别 | 典型场景 | 目标 |
|----------|----------|------|
| 空间分析 | 叠加分析、缓冲区分析、网络分析 | 支撑 GIS 业务计算 |
| 数据质检 | 拓扑检查、属性完整性校验、空间一致性验证 | 提升数据质量 |
| 统计报表 | 土地利用统计、人口密度分析、设施覆盖率 | 形成决策支撑输出 |
| 数据转换 | 格式互转、坐标系转换、数据融合 | 支撑数据治理 |
| 批量处理 | 大批量文件质检、自动化入库检查 | 提升自动化能力 |

> **说明**：以上场景的支撑深度取决于算子体系的丰富程度。框架本身提供的是可扩展的执行底座，具体算子的覆盖范围将在核心框架（Phase 1，见第 9 节）稳定后，随后续迭代逐步增强。

## 5. 业务对象定义

### 5.1 分析规则（Operator / 算子）

分析规则（以下简称"算子"）是可复用的最小处理能力单元，是框架中不可再分的原子操作。一个算子需要具备三个层次的完整定义：

**一、入口定义（输入可配置）**

算子的输入应具备可配置性——同一个算子可作用于不同的数据源和不同的目标字段。例如，一个"字段长度检查"算子不应被绑定到某个具体的 Shapefile 或某个固定的字段上，而是应该允许在使用时动态指定：在哪个数据源的哪个字段上执行检查。入口定义包含：

- 数据来源（适配到具体的数据源）
- 目标字段或要素（要处理的具体数据子集）
- 执行所需参数

**二、执行过程定义（处理逻辑明确）**

算子的核心处理逻辑是固定的（如检查字段长度、判断拓扑关系、执行空间叠加等），其实现可以是 SQL 查询、空间运算、属性比对等任意形式，但对外应暴露统一的可执行接口。

**三、输出定义（输出自适应场景）**

同一算子的输出形式应根据使用场景而变化：

- 在**质检场景**中：输出"数据 X 的字段 Y 不符合要求"等具体问题描述，供问题清单使用
- 在**分析场景**中（作为分析流程的一个步骤）：可能只需输出 `true` / `false` 或一个中间数据集，供下游分析项消费
- 在**报表场景**中：可能输出统计值、汇总信息等

**算子管理要求：**

- 可注册：算子能被框架发现并注册到算子池
- 可发现：可通过分类、名称、标签等方式检索算子
- 可扩展：支持通过标准接口开发自定义算子并接入框架

### 5.2 分析项（Analysis Item）

分析项是算子在具体业务场景下的**实例化配置**——将算子绑定到明确的数据输入、参数和输出目标后，即形成一个可直接执行的分析项。每一个分析项都是一个具体的、可独立执行的最小业务单元。

一个分析项应明确：
- 所引用的算子（来自算子池）
- 输入数据来源（明确到具体的数据源、图层、字段或要素集）
- 运行参数（算子的具体参数值）
- 输出目标与格式（结果写入哪里、以什么格式组织）

**重要约束：**

- 多个分析项可组合成一个分析方案（见 5.3）
- **分析项不支持嵌套**：即一个分析项内部不能再包含子分析项。这是为了控制复杂度，避免出现指数级的嵌套关系。若确需多层组合，应通过分析方案的子方案嵌套（见 5.3(2)）来实现

### 5.3 分析方案（Analysis Plan）

分析方案是面向业务目标的完整执行单元，由多个分析项按一定的执行关系组合而成，经过统一的调度执行后产出最终结果。一个方案应支持以下能力：

**（1）多分析项组合**

一个方案可包含任意数量的分析项，通过配置将它们组织在一起，形成完整的业务处理流程。

**（2）子方案嵌套**

方案之间支持嵌套——一个方案可以引用另一个完整的子方案作为其组成部分，从而实现流程的复用和分层。子方案嵌套是实现复杂流程的重要手段，但由于其实现难度较高（涉及递归解析、上下文隔离、结果回传等），**建议安排在 Phase 3（扩展优化阶段，见第 9 节）实现**，优先完成串并联组合等核心调度能力。

> **注意**：分析项（5.2）不支持嵌套，但分析方案之间可以嵌套。若某一方案需要多层级的处理逻辑，应通过引用子方案来组织，而非在单个分析项内部套嵌。

**（3）串并联执行关系**

方案中的分析项支持串行和并行两种基本的执行关系，不引入更复杂的工作流模式（如条件分支、循环审批等），以保持实现的可控性。

典型示例：假设一个方案包含 7 个分析项（①~⑦）：
- **串联组**：① → ② → ③（严格按顺序执行，后一项依赖前一项的结果）
- **并联组**：④、⑤、⑥ 可同时执行（相互之间无数据依赖）
- **汇合点**：⑦ 在 ④、⑤、⑥ 全部完成后才能执行

这种"串联组 → 并联组 → 汇合"的模式是最常见的执行拓扑，框架优先保障对此模式的良好支持。

**（4）数据流依赖定义**

当一个方案中包含多个分析项时，上游分析项产出的结果需要被下游分析项消费，因此框架必须在方案层面提供**数据流依赖的定义能力**，即明确各分析项之间的数据传递关系。具体需求包括：

- 支持显式声明分析项之间的输入-输出依赖关系
- 支持中间数据的格式适配与传递（上游输出格式可能与下游期望的输入格式不同）
- 支持数据依赖的完整性校验（确保下游所需的输入在上游确已产出）

> **实现思路参考**（非需求约束，供设计阶段讨论）：可考虑在存在数据依赖的两个分析项之间引入轻量的"数据中转节点"（隐式配置或专门的结合项），负责将上游输出转化为下游可消费的格式，以此降低实现复杂度。

**（5）模板化**

模板化是指为某类业务方向提供预置的分析方案模板。使用者只需提供符合规范的数据源，即可直接获得该业务方向的分析结果，无需自行配置方案。

模板化的**前提条件**：输入数据的目录结构、组织结构必须遵循模板预设的格式规范。只有数据在组织层面是确定的时候，模板化才有实际意义。若数据结构任意变化，则必须回到自定义方案的路径。

**（6）版本化管理**

版本化涉及两个维度：

- **方案版本化**：对方案进行修改后产生新版本，并支持回溯到任意历史版本（类似版本控制理念）。例如，方案 A 被修改为方案 B 后，若发现方案 B 效果不佳，应能方便地回退到方案 A。
- **执行结果版本化**：在同一方案下，对不同时期的数据（如不同月份、不同年份的数据集）执行分析，会产生不同版本的结果。这些结果版本之间可能存在比较需求（如数据变化监测、同比环比分析等），因此框架需要对执行结果进行版本标识和管理。

## 6. 功能需求

### 6.1 方案管理需求

框架应支持：

- 方案创建、加载、保存和更新
- 方案配置校验
- 方案模板管理
- 方案版本管理
- 方案导入导出

### 6.2 算子管理需求

框架应支持：

- 内置通用 GIS 算子
- 按分类管理算子
- 查询算子元数据与参数定义
- 通过标准接口扩展自定义算子
- 管理算子版本与兼容性

内置算子至少覆盖以下类别：

- 空间关系
- 空间运算
- 属性操作
- 空间连接
- 统计分析
- 格式转换
- 质检规则

### 6.3 数据源接入需求

框架应支持以下数据来源：

- PostgreSQL / PostGIS
- Shapefile
- GDB
- GeoJSON
- REST API / WFS 等远程服务

同时应满足：

- 屏蔽数据源差异
- 支持空间参考声明
- 支持大数据量读取
- 支持新增适配器扩展

### 6.4 输出管理需求

框架应支持将结果输出到：

- 数据库表
- 文件（Shapefile、GeoJSON、GeoPackage、CSV 等）
- 控制台或接口响应

并支持：

- 输出目标配置
- 输出字段选择
- 输出格式控制

### 6.5 执行与调度需求

框架应支持：

- 根据数据依赖自动确定执行顺序
- 串行与并行混合执行
- 方案级并发度控制
- 状态跟踪和执行监控
- 执行超时、取消和优雅关闭

### 6.6 质检能力需求

框架应支持将质检视为分析场景的特化能力，并满足：

- 质检规则执行
- 中间结果保留
- 细粒度执行日志
- 问题清单输出
- 差异对比能力
- 数据血缘与溯源能力
- 质量评分能力

至少应覆盖以下质检规则：

- 拓扑检查
- 属性完整性检查
- 属性一致性检查
- 空间一致性检查
- 几何有效性检查
- 重复检查
- 精度检查

### 6.7 报告与追溯需求

框架应支持输出：

- 分析结果
- 质检报告
- 问题明细
- 执行日志
- 血缘信息
- 统计汇总信息

## 7. 技术与实现约束

- 技术栈基于 C# / .NET 10
- GIS 能力基于 [OpenGIS Utils for .NET](https://github.com/znlgis/opengis-utils-for-net) 
- 方案配置以 JSON Schema 为主
- 依赖注入采用 Microsoft.Extensions.DependencyInjection
- 日志采用 Microsoft.Extensions.Logging
