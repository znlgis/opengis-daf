# GIS 数据分析与质检框架需求文档

> 对应设计文档：[/docs/GIS数据分析框架设计文档.md](./GIS数据分析框架设计文档.md)

## 1. 文档说明

本文档聚焦框架的业务目标、范围、核心概念、能力需求和约束条件，用于指导后续架构设计、开发排期和验收。  
关于技术实现细节、接口模型和执行机制，请参阅配套的《GIS数据分析框架设计文档》。

## 2. 项目背景

本项目旨在构建统一的 GIS 数据分析与数据质检框架。  
数据分析与数据质检在本质上都围绕“配置方案、执行处理、产出结果”展开，因此框架需要在一套统一能力底座之上同时支撑分析场景和质检场景，避免重复建设。

## 3. 建设目标

### 3.1 总体目标

- 建立统一的 GIS 数据分析与质检能力底座
- 通过配置化方案支持不同业务流程快速组合
- 支持多种数据源、多种输出方式和可扩展算子体系
- 为后续服务化、平台化和自动化集成打下基础

### 3.2 核心理念

无论是数据分析还是数据质检，其最根本的执行单元都可以抽象为一个原子化模型：**输入数据 → 执行操作 → 产出结果**。一个算子（Operator）就是这样一个原子操作的封装——接收输入、执行逻辑、返回输出。

当需要同时处理一组数据或串联一系列操作时，仅仅有孤立的算子是不够的，这时便产生了"方案"的概念。方案是多个算子实例化后（即分析项）按一定执行关系组织起来的有序集合，经框架统一调度执行后产出一个完整的业务结果（如质检报告、分析结果集、统计报表等）。

分析场景与质检场景之间的本质关系是理解本框架的关键：**质检是分析的一种特化场景**。二者的核心差异不在于技术执行层面，而在于关注点的不同——分析重在"结果是什么"，质检重在"过程哪里有问题"。因此，框架只需在一套执行引擎之上，为质检场景额外叠加过程追溯、问题清单和中间结果保留等能力即可。

基于以上认识，框架遵循以下核心理念：

- **方案驱动**：以分析方案为核心组织业务流程，方案可配置、可复用，支持通过子方案引用的方式实现分层组合（子方案嵌套机制将在后续迭代中完善，详见 5.3(2)）
- **统一框架**：分析和质检共用同一套执行框架，通过算子复用和输出适配，避免为两种场景分别建设独立系统
- **能力扩展**：通过算子体系实现能力扩展，算子需遵循统一接口，支持插件式注册和自定义扩展
- **过程可追溯**：尤其在质检场景下，保留过程信息、中间结果和执行日志，支撑问题溯源和结果复核

## 4. 应用场景

| 场景类别 | 典型场景 | 目标 |
|----------|----------|------|
| 空间分析 | 叠加分析、缓冲区分析、网络分析 | 支撑 GIS 业务计算 |
| 数据质检 | 拓扑检查、属性完整性校验、空间一致性验证 | 提升数据质量 |
| 统计报表 | 土地利用统计、人口密度分析、设施覆盖率 | 形成决策支撑输出 |
| 数据转换 | 格式互转、坐标系转换、数据融合 | 支撑数据治理 |
| 批量处理 | 大批量文件质检、自动化入库检查 | 提升自动化能力 |

> **说明**：以上场景的支撑深度取决于算子体系的丰富程度。框架本身提供的是可扩展的执行底座，具体算子的覆盖范围将在核心框架（Phase 1，见第 9 节）稳定后，随后续迭代逐步增强。

## 5. 业务对象定义

### 5.1 分析规则（Operator / 算子）

分析规则（以下简称"算子"）是可复用的最小处理能力单元，是框架中不可再分的原子操作。一个算子需要具备三个层次的完整定义：

**一、入口定义（输入可配置）**

算子的输入应具备可配置性——同一个算子可作用于不同的数据源和不同的目标字段。例如，一个"字段长度检查"算子不应被绑定到某个具体的 Shapefile 或某个固定的字段上，而是应该允许在使用时动态指定：在哪个数据源的哪个字段上执行检查。入口定义包含：

- 数据来源（适配到具体的数据源）
- 目标字段或要素（要处理的具体数据子集）
- 执行所需参数

**二、执行过程定义（处理逻辑明确）**

算子的核心处理逻辑是固定的（如检查字段长度、判断拓扑关系、执行空间叠加等），其实现可以是 SQL 查询、空间运算、属性比对等任意形式，但对外应暴露统一的可执行接口。

**三、输出定义（输出自适应场景）**

同一算子的输出形式应根据使用场景而变化：

- 在**质检场景**中：输出"数据 X 的字段 Y 不符合要求"等具体问题描述，供问题清单使用
- 在**分析场景**中（作为分析流程的一个步骤）：可能只需输出 `true` / `false` 或一个中间数据集，供下游分析项消费
- 在**报表场景**中：可能输出统计值、汇总信息等

**算子管理要求：**

- 可注册：算子能被框架发现并注册到算子池
- 可发现：可通过分类、名称、标签等方式检索算子
- 可扩展：支持通过标准接口开发自定义算子并接入框架

### 5.2 分析项（Analysis Item）

分析项是算子在具体业务场景下的**实例化配置**——将算子绑定到明确的数据输入、参数和输出目标后，即形成一个可直接执行的分析项。每一个分析项都是一个具体的、可独立执行的最小业务单元。

一个分析项应明确：
- 所引用的算子（来自算子池）
- 输入数据来源（明确到具体的数据源、图层、字段或要素集）
- 运行参数（算子的具体参数值）
- 输出目标与格式（结果写入哪里、以什么格式组织）

**重要约束：**

- 多个分析项可组合成一个分析方案（见 5.3）
- **分析项不支持嵌套**：即一个分析项内部不能再包含子分析项。这是为了控制复杂度，避免出现指数级的嵌套关系。若确需多层组合，应通过分析方案的子方案嵌套（见 5.3(2)）来实现

### 5.3 分析方案（Analysis Plan）

分析方案是面向业务目标的完整执行单元，由多个分析项按一定的执行关系组合而成，经过统一的调度执行后产出最终结果。一个方案应支持以下能力：

**（1）多分析项组合**

一个方案可包含任意数量的分析项，通过配置将它们组织在一起，形成完整的业务处理流程。

**（2）子方案嵌套**

方案之间支持嵌套——一个方案可以引用另一个完整的子方案作为其组成部分，从而实现流程的复用和分层。子方案嵌套是实现复杂流程的重要手段，但由于其实现难度较高（涉及递归解析、上下文隔离、结果回传等），**建议安排在 Phase 3（扩展优化阶段，见第 9 节）实现**，优先完成串并联组合等核心调度能力。

**（3）串并联执行关系**

方案中的分析项支持串行和并行两种基本的执行关系，不引入更复杂的工作流模式（如条件分支、循环审批等），以保持实现的可控性。

典型示例：假设一个方案包含 7 个分析项（①~⑦）：
- **串联组**：① → ② → ③（严格按顺序执行，后一项依赖前一项的结果）
- **并联组**：④、⑤、⑥ 可同时执行（相互之间无数据依赖）
- **汇合点**：⑦ 在 ④、⑤、⑥ 全部完成后才能执行

这种"串联组 → 并联组 → 汇合"的模式是最常见的执行拓扑，框架优先保障对此模式的良好支持。

**（4）数据流依赖定义**

当一个方案中包含多个分析项时，上游分析项产出的结果需要被下游分析项消费，因此框架必须在方案层面提供**数据流依赖的定义能力**，即明确各分析项之间的数据传递关系。具体需求包括：

- 支持显式声明分析项之间的输入-输出依赖关系
- 支持中间数据的格式适配与传递（上游输出格式可能与下游期望的输入格式不同）
- 支持数据依赖的完整性校验（确保下游所需的输入在上游确已产出）

> **实现思路参考**（非需求约束，供设计阶段讨论）：可考虑在存在数据依赖的两个分析项之间引入轻量的"数据中转节点"（隐式配置或专门的结合项），负责将上游输出转化为下游可消费的格式，以此降低实现复杂度。

**（5）模板化**

模板化是指为某类业务方向提供预置的分析方案模板。使用者只需提供符合规范的数据源，即可直接获得该业务方向的分析结果，无需自行配置方案。

模板化的**前提条件**：输入数据的目录结构、组织结构必须遵循模板预设的格式规范。只有数据在组织层面是确定的时候，模板化才有实际意义。若数据结构任意变化，则必须回到自定义方案的路径。

> 模板管理的具体操作需求（保存、创建、导入导出等）详见 §6.1(5)。

**（6）版本化管理**

版本化涉及两个维度：

- **方案版本化**：对方案进行修改后产生新版本，并支持回溯到任意历史版本。具体管理方式详见 §6.1(4)。
- **执行结果版本化**：在同一方案下，对不同时期的数据（如不同月份、不同年份的数据集）执行分析，会产生不同版本的结果。这些结果版本之间可能存在比较需求（如数据变化监测、同比环比分析等），框架需要对执行结果进行版本标识和管理。算子/插件的版本管理详见 §6.2(4)。

## 6. 功能需求

### 6.1 方案管理需求

框架应提供完整的方案生命周期管理能力。

**（1）方案基本操作**

- 方案创建：支持从零开始创建空白方案，或基于已有模板快速创建
- 方案加载与保存：支持将方案配置持久化为 JSON 文件，并从文件反序列化加载
- 方案更新：支持对已有方案进行修改并保存，包括添加/删除/调整分析项、修改参数配置等
- 方案复制：支持一键复制已有方案，在此基础上修改形成新方案，提高方案复用效率
- 方案导入导出：支持方案在不同环境或用户之间迁移，导出为独立可交换的配置文件

**（2）方案组织管理**

- 目录组织：支持以文件夹层级结构组织方案文件，不限制嵌套深度（但建议控制在 3 层以内以避免管理复杂度过高）。文件夹本身仅为组织手段，不承载业务语义
- 方案分组：支持按项目、业务方向或使用场景将方案归入不同分组，便于检索和批量管理
- 方案命名规范：方案应以有意义的名称标识，支持中文命名，建议包含业务场景和版本信息

**（3）方案配置校验**

- 结构校验：基于 JSON Schema 校验方案配置文件的结构合法性
- 业务规则校验：检查算子引用是否存在、输入输出绑定是否完整、数据依赖是否构成有向无环图（DAG）、参数是否在合法范围内等
- 预执行校验：在实际执行前尽可能发现配置问题，避免运行到一半才报错

**（4）方案版本管理**

- 版本标识：方案修改后产生新版本，通过版本号区分（如 V1、V2，或日期版本号如 20240604、20240605）
- 版本回溯：支持查看和回退到任意历史版本，以文件副本方式实现（复制方案文件并附加版本后缀）
- 版本对比：支持对比不同版本之间的差异，便于追踪方案演变

**（5）模板化管理**

为降低方案配置门槛，框架应提供模板化管理能力（概念说明参见 §5.3(5)）：

- 方案模板保存：将经过验证、效果良好的方案保存为模板，模板仅定义处理逻辑和参数约束，不绑定具体数据源
- 基于模板创建方案：使用者选择模板后，只需指定数据源即可快速生成可执行方案，无需从零配置
- 模板的前提条件：模板化要求输入数据在组织结构上符合模板预设的格式规范（如字段名、数据类型、目录结构等）。若数据结构任意变化，则必须回到自定义方案的路径
- 模板管理操作：模板作为特殊类型的方案文件，存放在独立的模板目录中，支持与普通方案相同的导入导出和版本管理能力
- 模板共享：支持模板的导入导出，便于团队内部分享和复用成熟的分析方案

### 6.2 算子与插件管理需求

框架应同时支持内置算子和外部插件两种扩展方式，统一纳入算子池管理。

**（1）算子分类与组织**

内置算子至少覆盖以下类别，并支持按类别检索和筛选：

| 类别 | 说明 | 示例算子 |
|------|------|----------|
| 空间关系 | 判断要素间的空间关系 | 相交判断、包含判断、相邻判断、距离判断 |
| 空间运算 | 对要素执行空间计算 | 缓冲区分析、叠加分析、裁剪、合并、差集 |
| 属性操作 | 对要素属性执行计算或变换 | 字段计算、属性映射、条件赋值、空值填充 |
| 空间连接 | 将两个数据集的要素按空间关系关联 | 空间关联、最近邻连接 |
| 统计分析 | 对数据集执行统计聚合 | 分组统计、密度分析、热点分析 |
| 格式转换 | 在不同数据格式或坐标系之间转换 | 格式互转、坐标系转换、几何类型转换 |
| 质检规则 | 数据质量检查专用规则 | 拓扑检查、属性完整性、几何有效性等（详见 6.6） |

**（2）算子元数据管理**

- 元数据查询：支持查询算子的名称、分类、功能描述、适用场景
- 参数定义：每个算子需声明其输入参数的名称、类型、是否必填、默认值、取值范围和约束条件
- 接口定义：每个算子需声明其输入 Schema（期望的要素属性结构）和输出 Schema（产出的要素属性结构）
- 版本信息：记录算子的版本号和兼容性说明

**（3）插件式扩展机制**

- 插件形态：自定义算子以 DLL 形式打包发布，遵循统一的 `IOperator` 接口规范
- 插件发现：算子通过主动导入的方式被发现和注册——用户在插件管理界面或配置文件中明确指定要加载的算子 DLL，框架在导入时通过反射发现实现了 `IOperator` 接口的类型，验证兼容性后注册到算子池（[设计决策变更] 原需求为启动时自动扫描目录，经安全评审后改为主动导入，避免自动扫描带来的恶意 DLL 加载风险）
- 插件隔离：不同插件加载到独立的 `AssemblyLoadContext` 中，避免版本冲突和类型污染，支持卸载
- 插件元数据标注：插件应通过注解（Attribute）或元数据文件声明自身信息——名称、用途、接口定义、参数说明、版本号等，供框架查询和展示

**（4）插件版本管理**

- 版本标识：插件通过 DLL 文件版本号或文件名后缀区分版本（如 `MyPlugin.V1.dll`、`MyPlugin.V2.dll`，或 `MyPlugin.20240604.dll`）
- 版本共存：同一插件的不同版本可同时存在于插件目录中，由方案配置指定使用哪个版本
- 兼容性声明：插件应声明其最低框架版本要求和已知的不兼容情况

**（5）插件组织管理**

- 目录组织：插件按文件夹层级组织（如 `plugins/spatial/`、`plugins/qc/`），支持嵌套但建议控制在 2 层以内
- 插件分组：可按功能领域将插件归入不同分组（如"空间分析插件包"、"质检规则插件包"、"数据转换插件包"），便于按需加载和管理
- 插件信息查询：框架提供插件查询接口，可按名称、分类、标签等条件检索已注册的插件，查看其元数据和参数定义

### 6.3 数据源接入需求

框架应通过统一的数据源适配层屏蔽不同数据来源的差异，为上层算子和执行引擎提供一致的要素读取接口。

**（1）关系型数据库**

以 PostgreSQL + PostGIS 为主要支持目标，兼顾其他关系型数据库的扩展能力：

- PostgreSQL / PostGIS：优先支持，作为空间数据存储和分析的主力数据库
- Oracle Spatial：提供适配支持（使用率正在降低，但存量系统仍有需求）
- MySQL：支持基础的空间数据读取（MySQL 8.0+ 的空间能力）
- SQL Server：提供适配支持

连接配置统一通过连接字符串管理，包含以下核心信息：服务器地址、端口、数据库名称、用户名、密码。敏感信息（如密码）应支持加密存储。

**（2）文件型数据源**

- **Shapefile**：最常用的矢量数据交换格式之一，必须优先支持。注意 Shapefile 的固有限制（如字段名长度限制、不支持多几何类型混合等），适配器需妥善处理
- **GDB（File Geodatabase / Enterprise Geodatabase）**：File GDB 必须支持；Enterprise GDB（通过 SDE 连接）作为可选的扩展能力
- **GeoJSON**：Web 端最常用的空间数据交换格式，对前端友好，支持直接读取和输出

**（3）服务型数据源**

- **REST API / Web API**：支持通过 HTTP 接口获取空间数据。各服务提供方的 API 格式各不相同，适配器需将外部数据结构转换为框架内部统一的数据模型。需注意此类数据源的可控性较低（权限、限流、数据结构变化等），建议主要用于内部系统间数据交换，不推荐在关键生产流程中强依赖外部 API
- **OGC WFS**：支持 WFS 标准协议。WFS 理论上支持数据编辑，但考虑到权限控制和安全审计的复杂性，框架优先支持 WFS 的读取能力，写入能力由上层业务系统自行控制
- **OGC 其他服务**（WMS、WMTS 等）：作为可选的扩展方向，根据业务需要逐步支持

**（4）数据源差异屏蔽**

- 统一要素模型：所有数据源适配器实现 `IFeatureSource` 接口，向上层暴露一致的要素读取能力，使算子无需关心底层数据来自数据库、文件还是服务
- 空间参考系统（CRS）声明：每个数据源必须声明其坐标参考系统，框架在执行前检查输入数据的 CRS 一致性；必要时通过 GDAL 内置 OSR 执行自动转换（NTS 场景下可选配 ProjNet）
- 字段映射：不同数据源的字段命名和类型体系各不相同，适配器负责将原始字段映射为框架内部统一的数据模型
- 过滤条件下推：尽可能将算子层面的过滤条件（空间范围过滤、属性过滤）下推到数据源层面执行，减少不必要的数据传输。例如，对 PostGIS 数据源，将空间查询条件转换为 SQL WHERE 子句

**（5）大数据量读取**

- 分页读取：支持按要素数量或空间范围分页读取，避免一次性加载全部数据导致内存溢出
- 流式读取：基于 `IAsyncEnumerable` 提供流式数据访问，数据按需逐条消费，不预先缓存全部结果
- 空间索引利用：大数据集查询优先使用数据源自带的空间索引（如 PostGIS 的 GiST 索引、Shapefile 的 .qix 索引）

**（6）中间数据格式**

- **WKT（Well-Known Text）**：作为几何数据在组件间传递的中间格式。WKT 以纯文本表示几何对象，适合在算子之间传递几何信息，无需依赖特定的二进制格式。WKT 不承载属性数据、坐标系等附加信息，仅作为几何信息的轻量交换格式
- 对于需要传递完整要素（几何 + 属性）的场景，使用框架内部的统一要素模型

**（7）适配器扩展**

- 框架提供标准的数据源适配器接口，支持第三方开发自定义适配器
- 新适配器以插件形式注册到框架，遵循与算子插件相同的发现和加载机制

**（8）几何类型支持**

框架应至少支持以下 OGC 标准几何类型，并保证所有算子能正确处理：

- Point（点）、MultiPoint（多点）
- LineString（线）、MultiLineString（多线）
- Polygon（面）、MultiPolygon（多面）
- GeometryCollection（几何集合）

对于不支持的几何类型（如 Curve、Surface 等），适配器应在读取时给出明确提示或尝试降级转换为支持的简单类型。

### 6.4 输出管理需求

框架应支持将分析或质检结果输出到多种目标，并允许灵活配置输出内容和格式。

**（1）输出目标类型**

| 输出目标 | 典型用途 | 说明 |
|----------|----------|------|
| 数据库表 | 持久化分析结果、供下游系统消费 | 支持 PostGIS、Oracle 等 |
| Shapefile | 数据交换、分发 | 注意字段名长度和几何类型限制 |
| GeoJSON | Web 前端展示、API 响应 | 对前端最友好的格式 |
| GeoPackage | 标准化的数据交换 | 单文件、支持多图层 |
| CSV | 统计报表、属性数据导出 | 不含几何信息 |
| 控制台 / API 响应 | 调试、实时查询 | 轻量级文本输出 |
| 对象存储（OBS / S3） | 生产环境的大文件存储 | 分布式、高可用、自动容灾 |

**（2）输出配置能力**

- 输出目标配置：每个分析项可独立指定输出目标（写入位置、文件路径、数据库连接等）
- 输出字段选择：支持选择输出的字段子集，而非全量输出所有属性
- 输出格式控制：支持配置数值精度、日期格式、字符编码等细节
- 中间结果输出：支持将计算过程中的中间结果也输出到指定位置（尤其在质检场景中，中间结果对问题溯源具有重要价值）

### 6.5 执行与调度需求

框架需要内置一个轻量级的调度引擎，负责方案中分析项的执行编排。该调度引擎定位为"简单调度框架"，而非完整的工作流引擎——不支持条件分支、循环审批、人工节点等复杂工作流模式。

**（1）执行关系定义**

- 串行执行：具有数据依赖关系的分析项按依赖顺序依次执行，后一项必须等待前一项完成
- 并行执行：相互之间无数据依赖的分析项可同时执行，充分利用多核计算资源
- 汇合等待：当某个分析项依赖多个上游结果时，调度器等待所有上游都完成后才触发该分析项执行

**（2）依赖分析与自动排序**

- 框架根据方案中分析项的输入输出绑定关系，自动分析数据依赖，构建有向无环图（DAG）
- 基于 DAG 的拓扑排序自动确定执行顺序——用户只需声明分析项之间的数据依赖关系，框架自动推导最优执行顺序，无需手动编排执行步骤的先后
- 循环依赖检测：在构建 DAG 时自动检测循环依赖，一旦发现即拒绝执行并给出明确的错误提示

**（3）并发控制**

- 方案级并发度控制：通过配置项 `MaxParallelism` 限制同一方案的并行分析项数量，避免过度消耗系统资源
- 全局并发度控制：支持限制整个框架的全局并行执行数，防止多个方案同时执行导致资源争抢
- 资源感知调度：未来可扩展为根据 CPU 核心数、可用内存等系统资源动态调整并发度

**（4）执行状态跟踪与监控**

- 状态机：每个分析项的执行状态遵循 `Pending → Queued → Executing → Success/Failed/Canceled` 生命周期
- 实时状态查询：支持随时查询方案的执行进度（已完成/执行中/待执行的分析项列表）
- 执行统计：记录整个方案的总耗时、各分析项的耗时、成功数、失败数等指标
- 失败重试：支持配置失败重试策略（最大重试次数、重试间隔），超阈值后进入最终失败状态

**（5）超时、取消与优雅关闭**

- 超时控制：分析项可配置执行超时时间，超时后自动终止并标记为失败
- 手动取消：支持用户主动取消正在执行的方案，通过 `CancellationToken` 向所有执行中的分析项传播取消信号
- 优雅关闭：收到关闭信号后，停止接受新任务、等待当前执行中的分析项完成（或超时强制终止）、持久化必要的中间结果和检查点，为后续恢复执行保留可能

**（6）性能需求**

框架在数据处理性能上应满足以下要求：

- 流式处理：基于 `IAsyncEnumerable` 实现数据按需逐条消费，避免一次性加载全量数据导致内存溢出
- 过滤条件下推：尽可能将算子层面的过滤条件（空间范围、属性筛选）下推到数据源层面执行，减少不必要的数据传输
- 结果缓存：支持对重复使用的中间计算结果进行缓存，避免相同输入条件下重复计算
- 数据分区并行：支持将大数据集按空间范围或属性分区，在各分区上并行执行同一分析项
- 增量计算（可选）：对于仅部分数据变更的场景，支持仅对变更部分重新计算，而非全量重跑

**（7）错误处理与错误码**

框架应建立统一的错误分类和错误码体系：

- 错误分类：将错误归为以下四类，便于日志检索、告警和自动化处理
  - **配置错误**：方案配置不合法（Schema 校验失败、算子不存在、参数越界等）
  - **数据源错误**：数据源连接失败、权限不足、数据格式异常等
  - **运行时错误**：算子执行过程中的未预期异常（内存不足、空引用等）
  - **数据错误**：个别数据记录不符合预期（几何无效、字段缺失等），此类错误通常不终止整个方案，仅影响当前要素
- 错误码格式：采用统一格式（如 `ERR_CATEGORY_SPECIFIC`），便于调用方编程处理
- 错误传播策略：
  - 配置错误 → 方案预校验阶段即拒绝，不进入执行
  - 数据源错误 → 终止整个方案执行
  - 运行时错误 → 按重试策略处理，超阈值后终止方案
  - 数据错误 → 仅跳过当前要素，继续处理后续要素（质检场景记录到问题清单）

### 6.6 质检能力需求

质检是分析场景的特化，二者共用同一套执行框架。质检能力通过在分析能力的基础上叠加以下特化能力来实现。

**（1）质检模式特性**

质检模式与普通分析模式的核心差异在于关注点的不同——分析关注"结果是什么"，质检关注"过程哪里有问题"。因此质检模式在普通分析执行引擎之上，额外叠加：

- **细粒度执行日志**：记录到要素级别的执行过程，标明每个要素的检查结果（通过/未通过/跳过），日志粒度可由方案配置控制
- **中间结果保留**：保留质检过程中的中间计算结果（如拓扑关系判断的中间几何、属性比对的时间值等），支撑后续的问题溯源和人工复核
- **问题清单输出**：将质检发现的问题以结构化清单形式输出，每条问题包含：问题编号、所属分析项、违规要素 ID、问题类型、严重级别、问题描述、相关上下文数据
- **差异对比**：支持原始数据与质检结果数据的差异对比（如原始数据集与清洗后数据集的差异），帮助定位数据质量问题
- **质量评分**：支持基于规则权重和问题统计给出综合质量评分（如 0-100 分），便于量化数据质量水平

**（2）数据溯源与版本追溯**

- 数据血缘：框架保留执行过程中"输入数据 → 处理步骤 → 输出结果"的关联关系。但考虑到完整血缘追溯的实现复杂度较高，框架优先支持简化版——以版本追溯代替完整血缘图谱
- 版本追溯：记录方案版本、算子版本、数据源版本和执行时间，形成可追溯的版本链。当发现质量问题时，可通过版本链定位到是方案变更、算子变更还是数据本身的问题
- 差异对比：不同版本执行结果的对比分析（如同一方案对 2023 年和 2024 年数据执行质检的结果对比），支撑数据质量趋势分析

**（3）质检规则覆盖范围**

框架至少应覆盖以下质检规则类别：

| 规则类别 | 检查目标 | 典型规则示例 |
|----------|----------|-------------|
| 拓扑检查 | 要素之间的空间关系是否正确 | 面不能重叠、线必须闭合、点必须在面内 |
| 属性完整性 | 必填字段是否有值 | 必填字段非空检查、字段值域范围检查 |
| 属性一致性 | 字段间逻辑是否一致 | 行政区划代码与名称匹配检查、面积与周长合理性 |
| 空间一致性 | 空间位置与属性描述是否一致 | 坐标落点是否在声称的行政区内 |
| 几何有效性 | 几何对象本身是否合法 | 自相交检查、环方向检查、退化几何检查 |
| 重复检查 | 是否存在重复要素 | 完全重复、几何重复（属性不同）、属性重复（几何不同） |
| 精度检查 | 数据精度是否满足要求 | 坐标精度、面积精度、几何节点密度 |

**（4）质检与分析的统一**

如 §3.2 所述，质检是分析的特化场景——同一算子可按方案配置在不同模式下运行。分析项标记为"质检模式"时，框架自动启用细粒度日志和中间结果保留（详见 §6.6(1)）；标记为"分析模式"时沿用默认执行策略。质检方案与分析方案使用完全相同的配置结构和执行引擎。

### 6.7 报告与追溯需求

框架应能根据执行过程和结果生成多层次的报告和追溯信息。

**（1）结果输出**

- 分析结果：按方案配置将分析产出的数据写入指定目标（详见 §6.4 输出管理需求）
- 质检报告：以结构化格式（JSON、HTML、PDF 等）输出质检报告，包含总体评分、规则级统计、问题清单等
- 问题明细：针对质检场景，输出每条问题的详细描述，包含问题位置、违规要素 ID、问题类型、严重级别和建议修复方式

**（2）执行日志**

- 可配置日志粒度：支持按方案级、分析项级、要素级三个粒度控制日志输出。分析场景默认使用分析项级，质检场景默认使用要素级
- 日志内容：每条日志应包含时间戳、所属分析项、日志级别、消息内容和可选的上下文数据（如当前处理的要素 ID）
- 日志输出目标：支持输出到控制台、文件、数据库或接入外部日志系统（详见 §7.4 日志与可观测性约束）
- 日志持久化：执行日志应持久化保存，支持事后查询和审计

**（3）统计汇总**

框架应在每次执行完成后自动生成统计汇总信息：

- 执行概况：执行开始时间、结束时间、总耗时、执行的方案名称和版本
- 分析项统计：每个分析项的耗时、处理要素数量、成功/失败/跳过数量
- 质检统计：总问题数、各级别问题数量（错误/警告/提示）、各类别问题数量、综合质量评分
- 资源使用统计：内存峰值、CPU 使用率（可选）、数据读取量（可选）

> **关于"数据血缘"的简化处理**：完整的血缘图谱（追溯每个输出要素由哪些输入要素经过哪些变换而来）实现复杂且计算成本高。框架现阶段以版本追溯代替完整血缘——记录方案版本 + 算子版本 + 数据源版本 + 执行时间的关键组合，足以满足多数追溯场景的需求。若业务确实需要要素级血缘，可在后续迭代中作为专项能力扩展。

### 6.8 安全需求

框架应满足以下安全约束，保障数据和系统的安全运行：

**（1）敏感信息保护**

- 连接信息加密存储：数据库连接字符串、API 密钥等敏感信息支持加密存储，不以明文方式持久化
- 方案配置中的敏感字段（如密码、Token）支持脱敏展示

**（2）数据源访问控制**

- 数据源访问受权限控制，框架应支持对接外部权限系统或提供基础的访问凭据管理
- 不同用户/角色对同一数据源可拥有不同的读写权限

**（3）API 层安全**

- 当框架以服务化形态部署时，API 层应支持认证（如 API Key、JWT、OAuth2）
- 支持请求限流，防止恶意调用或意外过载
- 对输入参数进行校验，防止注入攻击和非法参数

**（4）审计日志**

- 关键操作（方案创建/修改/删除、执行启动/取消/完成、数据源连接等）应纳入审计日志
- 审计日志包含操作者、操作时间、操作类型、操作对象和操作结果
- 审计日志不可篡改，与普通执行日志分离存储

## 7. 技术与实现约束

### 7.1 技术栈选型

| 技术要素 | 选型 | 说明 |
|----------|------|------|
| 开发语言 | C# 14 | 与 .NET 10 配套，利用最新语言特性（record、pattern matching、primary constructors 等）提升代码可维护性 |
| 运行框架 | .NET 10 | 最新 LTS，跨平台，长期支持，支持 NativeAOT 编译 |
| GIS 内核 | GDAL/OGR | C++ 内核，跨平台（含 Linux ARM、国产环境），内置坐标转换，原生支持主流矢量/栅格格式；大数据场景性能优于纯 .NET 方案。通过 [OpenGIS Utils for .NET](https://github.com/znlgis/opengis-utils-for-net) 封装，规避直接调用 C API 的跨平台互操作复杂性 |
| GIS 补充 | NetTopologySuite（NTS） | 轻量纯 .NET 空间计算库，作为 GDAL 的补充用于简单场景（几何模型构建、空间索引、轻量空间运算） |
| 坐标转换 | GDAL 内置 OSR | GDAL 适配器使用内置 OSR 进行 CRS 转换；NTS 场景下可选配 ProjNet |
| 方案配置 | JSON Schema | 便于校验、生成工具支持好、可读性强 |
| 依赖注入 | Microsoft.Extensions.DependencyInjection | 与 .NET 生态一致，支持插件注册和生命周期管理 |
| 日志 | Microsoft.Extensions.Logging | 统一的日志抽象层，支持通过 Provider 机制接入不同的日志实现（控制台、文件、Seq、Elasticsearch 等） |
| 序列化 | System.Text.Json | 方案配置与元数据的序列化与反序列化 |

> **关于 GIS 内核的选型说明**：框架以 GDAL/OGR 为核心 GIS 内核，通过 OpenGIS Utils for .NET 封装层接入。GDAL 优势在于：(1) 跨平台覆盖更广，包括 ARM 和国产 Linux 环境；(2) 内置坐标转换 OSR，无需额外依赖；(3) 原生支持 Shapefile、GDB、PostGIS 等主流格式；(4) 大数据量场景下 C++ 内核性能更优。
>
> NTS（NetTopologySuite）保留作为轻量补充——在纯 .NET 环境下的简单空间运算、几何模型构建和空间索引场景中提供更便捷的 API。数据通过 GDAL 流入框架后，根据算子需求在 GDAL 原生计算和 NTS 计算之间选择最优路径，结果再通过 GDAL 或其他输出适配器写出。这种双引擎设计兼顾了大数据场景的性能和小数据场景的开发效率。

### 7.2 插件架构约束

框架通过 DLL 插件机制实现算子能力扩展，需遵循以下约束：

- **标准接口规范**：所有自定义算子插件必须实现 `IOperator` 接口，数据源插件实现 `IFeatureSource`，输出插件实现 `IFeatureSink`
- **插件发现机制**：算子通过主动导入的方式被发现和注册——用户在插件管理界面或配置文件中明确指定要加载的算子 DLL，框架在导入时通过反射发现实现了 `IOperator` 接口的类型并验证兼容性。此设计将"哪些算子可用"的控制权交给使用者，避免自动扫描带来的管理混乱和安全风险（[设计决策变更] 原需求为启动时自动扫描，经安全评审后改为主动导入）
- **插件隔离**：使用 `AssemblyLoadContext` 实现插件的独立加载和隔离，避免不同插件之间的类型冲突，支持卸载
- **依赖注入集成**：插件通过 DI 容器注册其服务，框架负责解析和注入插件所需的依赖项（如日志、配置、数据源等）
- **元数据标注**：插件应通过 Attribute 或元数据文件声明以下信息：名称、版本号、接口实现类型、功能描述、参数定义。这些元数据供框架查询和展示，支撑算子检索和方案配置校验
- **版本兼容**：同一插件的不同版本（通过 DLL 文件名或 Assembly 版本区分）可共存于插件目录，方案配置可指定使用特定版本

### 7.3 存储与持久化约束

- **方案存储**：方案配置以 JSON 文件为主要持久化形式，文件组织采用文件夹层级结构。这种方式简单直观、便于版本管理（直接复制文件即可实现版本分支），且无需依赖数据库
- **方案模板存储**：模板与方案使用相同的 JSON 文件格式，存放于独立的模板目录中
- **执行日志存储**：执行日志可输出到文件、数据库或外部日志系统，具体由日志 Provider 配置决定
- **对象存储支持（可选）**：生产环境中，大文件输出（如质检报告、导出的数据集）可接入对象存储（OBS、S3 兼容存储），提供分布式高可用和自动容灾能力。本地文件存储仅适用于开发和单机部署场景——磁盘故障可能导致数据不可恢复
- **中间结果存储**：执行过程中的中间结果优先使用内存缓存，必要时溢出到临时文件。质检场景的中间结果需按配置决定是否持久化保留

### 7.4 日志与可观测性约束

- **日志框架**：基于 `Microsoft.Extensions.Logging` 抽象层，不绑定具体的日志实现。使用者可根据需要选择不同的日志 Provider（控制台、文件、Seq、Elasticsearch、Application Insights 等）
- **结构化日志**：日志采用结构化格式（如 JSON），便于日志系统索引、检索和分析
- **日志粒度分级**：
  - 方案级：记录方案开始/结束、总体统计
  - 分析项级（默认）：记录每个分析项的执行开始/结束、耗时、状态
  - 要素级（质检模式默认）：记录每个要素的处理结果和详情
- **日志标签**：每条日志应携带方案 ID、执行 ID、分析项 ID 等上下文标签，便于跨日志系统的关联查询
- **可观测性扩展**：框架预留 OpenTelemetry 集成点，支持导出 Trace、Metrics 和 Logs 到 Prometheus、Grafana、Jaeger 等可观测性平台

### 7.5 平台兼容性约束

- **跨平台运行**：框架及其依赖（GDAL、NTS、.NET 10）需能在以下平台上编译和运行：
  - Windows（x64）
  - Linux（x64、ARM64）
  - macOS（x64、ARM64）
- **ARM 兼容性**：GDAL 的关键优势之一是对 ARM 架构的良好支持，通过 OpenGIS Utils for .NET 封装层规避了直接编译 C/C++ 库在 ARM 环境下的兼容性问题
- **容器化部署**：框架需支持 Docker 容器化部署，提供官方 Docker 镜像或 Dockerfile 模板
- **无 UI 依赖**：核心框架不依赖任何 GUI 组件，纯控制台/服务化运行

### 7.6 扩展性与开闭原则约束

- **对扩展开放**：算子、数据源适配器、输出适配器均通过标准接口扩展，第三方可以独立开发并发布插件
- **对修改封闭**：核心执行引擎、调度器和方案管理器的内部实现不因新增算子或适配器而修改
- **依赖注入驱动**：所有组件通过 DI 容器组装，新增组件只需注册到容器即可被框架发现和使用
- **配置优先于代码**：方案的行为差异应尽量通过配置（JSON Schema）表达，而非通过修改框架代码。只有在配置无法表达的能力（如全新的算法逻辑）才需要开发新算子
- **开源与社区友好**：框架核心以开源形式发布（基于 OpenGIS Utils for .NET 生态），鼓励社区贡献算子插件和数据源适配器。框架本身保持轻量，不捆绑大量非核心功能

## 8. 非功能需求补充

### 8.1 性能需求

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 单算子处理速率 | ≥ 1,000 要素/秒 | 简单属性操作场景（如字段计算、空值填充） |
| 空间连接耗时 | ≤ 60 秒 | 100 万要素 × 100 万要素的空间连接（含索引构建） |
| 框架冷启动时间 | ≤ 3 秒 | 不含插件加载时间 |
| 内存峰值 | ≤ 2 GB | 百万级要素处理场景下的默认上限，可通过配置调整 |

### 8.2 可靠性需求

- **可用性目标**：框架核心执行链路可用性 ≥ 99.5%（排除数据源和网络故障）
- **故障恢复**：长时间运行的方案支持优雅关闭——接收到关闭信号后完成当前分析项后安全退出，不丢失已完成结果
- **部分成功策略**：批量处理场景中，单个分析项失败不中断整个方案（可配置失败策略为 ContinueIndependent）

### 8.3 容量需求

| 指标 | 目标值 |
|------|--------|
| 单方案最大分析项数 | 1,000 |
| 最大并发方案数 | 10（单机） |
| 插件目录最大 DLL 数 | 200 |
| 单数据集最大要素数 | 1,000 万（流式处理） |

### 8.4 向后兼容性

- **方案 JSON Schema 版本策略**：主版本号变更时可能引入不兼容变更；次版本号变更保证向后兼容。框架需支持声明目标 Schema 版本，并对不兼容方案给出明确升级指引
- **算子接口兼容**：`IOperator` 接口的主版本（如 1.x → 2.0）变更需提供迁移指南和弃用过渡期

## 9. 阶段规划

### Phase 1（MVP — 核心框架）

**目标**：实现可运行的端到端链路，验证核心架构可行性。

| 模块 | 范围 |
|------|------|
| 执行引擎 | 串联执行（按 DAG 拓扑顺序依次执行分析项） |
| 算子池 | 算子注册、元数据管理、主动导入机制 |
| 核心算子 | 5-8 个高频算子（缓冲区、相交判断、属性完整性检查、字段计算、格式转换等） |
| 数据源适配 | PostGIS、Shapefile、GeoJSON（共 3-5 个） |
| 输出适配 | PostGIS、Shapefile、GeoJSON、Console |
| 方案管理 | JSON Schema 校验、方案 CRUD、版本管理（文件副本） |
| 质检能力 | 基础质检模式（问题清单 + 质量评分） |
| 日志 | 结构化日志，方案级 + 分析项级 |
| 平台 | Windows x64 + Linux x64 |

**不包含**：并行调度、DAG 可视化、子方案嵌套、分布式、API 层、插件市场

### Phase 2（功能增强）

**目标**：多核算力利用、算子体系扩展、生产级适配器。

| 模块 | 新增范围 |
|------|----------|
| 执行引擎 | 并行调度（同拓扑层并行）、DAG 可视化导出 |
| 算子扩展 | 扩展至 20+ 算子（空间连接、叠加分析、统计聚合等） |
| 数据源适配 | Oracle Spatial、SQL Server、MySQL、WFS、InMemory |
| 输出适配 | GeoPackage、CSV、HTML 报告 |
| 质检增强 | 版本追溯对比（IVersionDiffService）、质量趋势分析 |
| 配置增强 | 方案模板管理、批量导入导出 |
| 安全 | API Key 认证、审计日志、敏感信息加密 |
| 性能 | 分区策略（IPartitionStrategy）、结果缓存（IResultCache） |

### Phase 3（生态完善）

| 模块 | 范围 |
|------|------|
| 调度引擎 | 子方案嵌套、动态并发调整、分布式调度（预留） |
| API/CLI | REST API 完整实现、CLI 工具 |
| 可视化 | Web 端轻量级流程编辑器 |
| 异构语言 | 进程外执行（gRPC sidecar），支持 Python/R 算子 |
| 插件市场 | 算子注册表、NuGet 标签体系、社区贡献流程 |
| 高级功能 | 栅格数据处理、要素级血缘（LineageRecord）、增量计算 |
